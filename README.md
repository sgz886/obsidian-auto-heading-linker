# Auto Heading Linker

An Obsidian plugin that automatically updates internal heading links (`[[Note#heading]]`) when headings are renamed — whether by manual editing, the Number Headings plugin, or any other means.

## Why

Obsidian updates `[[Note]]` links when you rename a file, but does **not** update `[[Note#heading]]` links when you edit a heading. The built-in "Rename this heading" (right-click) works but is manual. This plugin makes it fully automatic.

## Installation

### From source

```bash
git clone https://github.com/<your-username>/obsidian-auto-heading-linker.git
cd obsidian-auto-heading-linker
npm install
npm run build
```

Then install into your vault:

```bash
mkdir -p <your-vault>/.obsidian/plugins/auto-heading-linker
cp main.js manifest.json <your-vault>/.obsidian/plugins/auto-heading-linker/
```

### Quick install (already built)

If you already have the built `main.js` and `manifest.json`:

```bash
mkdir -p <your-vault>/.obsidian/plugins/auto-heading-linker
cp main.js manifest.json <your-vault>/.obsidian/plugins/auto-heading-linker/
```

### Enable the plugin

1. Open Obsidian
2. Go to **Settings → Community plugins**
3. If prompted, turn off **Restricted mode** (required for third-party plugins)
4. Click **Reload plugins** — "Auto Heading Linker" should appear in the list
5. Toggle it **on**

### File Descriptions

| File | Purpose |
|------|---------|
| `main.ts` | All plugin logic. Contains the `AutoHeadingLinker` class with snapshot, diff, and link update methods. This is the only file you need to edit for functionality changes. |
| `main.js` | Compiled output from `main.ts`. Generated by `npm run build`. Do not edit directly. |
| `manifest.json` | Required by Obsidian. Defines plugin id (`auto-heading-linker`), name, version, and minimum Obsidian version. |
| `package.json` | Defines the `build` script using esbuild. Dependencies: `obsidian` (type definitions), `esbuild` (bundler). |
| `tsconfig.json` | TypeScript config. Targets ES2020, uses ESNext modules. |
| `design.md` | Full design document covering architecture, data flow, API usage, limitations, and future improvements. **Read this first** if you want to understand or modify the plugin. |

## How It Works (Summary)

1. **Snapshot**: When you open a file, the plugin records all headings and their line numbers in memory
2. **Diff**: When the file changes, it compares new headings against the snapshot by line number
3. **Update**: For any heading that changed, it uses Obsidian's metadata cache to find all links pointing to the old heading and rewrites them

See `design.md` for the full detailed design.

## Key Methods in main.ts

| Method | Description |
|--------|-------------|
| `onload()` | Registers event listeners for file-open and metadata-changed |
| `takeSnapshot(file)` | Reads headings from metadata cache, stores as `{line, heading}[]` |
| `onFileChanged(file, cache)` | Entry point for change detection; calls diff then update |
| `diffHeadings(oldSnap, newHeadings)` | Compares old vs new headings by line number, returns renames |
| `updateLinks(targetFile, changes)` | Finds and rewrites all links across the vault for renamed headings |
| `resolveLink(linkText, sourceFile)` | Resolves a wikilink note part to a file path using metadata cache |

## Build & Development

```bash
# Install dependencies
npm install

# Build (compiles main.ts → main.js)
npm run build

# After building, restart Obsidian or run "Reload app without saving" (Cmd+Option+I → Console → app.commands.executeCommandById('app:reload'))
```

## Known Limitations

- **Cold start**: On Obsidian restart, no snapshot exists until you open a file. Heading changes before the first snapshot are missed.
- **Line-based matching**: If lines are inserted above a heading (shifting it down), the diff may not detect the rename until the next edit.
- See `design.md` §8 for full details.
